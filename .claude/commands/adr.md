당신은 PuppyTalk 백엔드의 아키텍처 의사결정을 ADR로 남긴다. 주제: **$ARGUMENTS**

## 근거 (반드시 먼저 읽고 인용 — 발명 금지)
- 운영 봉투·범위: @docs/00-operating-envelope-and-scope.md
- 횡단 아키텍처 결정·ADR 목록: @docs/01-architecture.md
- 관련 실제 코드를 열어 확인한다(주제에 해당하는 `app/domain/…` 또는 `app/core|infra|common`).
- 버그·최적화 근거가 필요하면 @analysis.md 의 항목번호를 인용한다.

## 절차
1. **주제 매핑**: `$ARGUMENTS` 를 `01-architecture.md` 하단 **ADR 목록**의 항목에 맞춘다.
   목록에 있으면 그 번호·제목을 쓰고, 없으면 목록 다음 번호를 새로 부여한다.
2. **번호 확정**: `docs/adr/` 의 기존 파일과 `01-architecture.md` 목록 중 **더 큰 번호 + 1** 로 충돌 없이 부여.
   파일명 = `docs/adr/NNNN-<kebab-slug>.md` (4자리, 예: `0002-cursor-pagination.md`).
3. **코드 확인**: 결정이 가리키는 실제 함수/파일 경로를 찾아 **관련 코드** 필드에 적는다(존재하는 것만).
4. **작성**: 아래 집 형식으로 `docs/adr/NNNN-….md` 를 만든다.
5. **인덱스**: `docs/adr/README.md` 가 있으면 목록 표에 한 줄 추가, 없으면 생성.
6. **보고**: 생성 경로 + 핵심 결정 1~2줄 요약. 커밋은 사용자 승인 후에만.

## 집 형식 (이 골격을 지킨다)

```markdown
# ADR NNNN — <제목>

- **상태**: 채택됨 (Accepted)
- **관련 코드**: `경로#심볼` (실제 존재하는 것만; 아직 없으면 "구현 예정")

## 맥락 (Context)
어떤 문제를, [운영 봉투](../00-operating-envelope-and-scope.md)의 **어떤 전제** 위에서 푸는가.
전제 없는 일반론 금지 — "초당 수백~수천 조회" 같은 봉투 수치를 근거로.

## 결정 (Decision)
무엇을 하기로 했는가. 번호 매긴 단계로 구체적으로.

## 트레이드오프 (Consequences)
**얻은 것** / **치른 비용** 을 각각 명시.

## 고려한 대안 (Alternatives)
| 대안 | 기각 사유 |
|------|-----------|
표로. 각 기각 사유는 봉투 전제나 구체 문제에 연결.

## 일부러 하지 않은 것 (Non-goals)
봉투 상한(멀티리전·강정합성·99.99%+·초당 수만 write)을 넘어 **의식적으로 배제**한 것.
이 필드가 *정당화된 복잡도*의 핵심 — 절대 비우지 않는다.
```

## 규칙
- 한국어로. 보일러플레이트 금지, 목적 있는 문장만.
- 봉투로 정당화되지 않는 복잡도는 ADR 대상이 아니다 — 그런 주제면 "이건 ADR 불필요"라고 알린다.
- 커밋 메시지·문서에 작업 지침 파일명을 언급하지 않는다("코드 리뷰"·"성능 최적화" 등 일반 표현).
