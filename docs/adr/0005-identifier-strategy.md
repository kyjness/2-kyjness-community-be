# ADR 0005 — 식별자 전략: UUID v7(PK) · Base62(공개) · ULID(토큰)

- **상태**: 채택됨 (Accepted)
- **관련 코드**: `app/core/ids.py`, `app/db/base_class.py`

## 맥락 (Context)

엔티티 PK와 외부 노출 ID를 정해야 한다. 순차 정수(auto-increment) PK는:

- **열거·추정 취약** — 공개 URL에 `/posts/123`이 노출되면 리소스 수·증가율을 추정당한다.
- **분산 생성 약점** — 멀티 인스턴스에서 충돌 없는 사전 생성이 어렵다.

단, 무작위 UUID v4를 PK로 쓰면 **인덱스 단편화**(삽입 위치가 랜덤)로 쓰기 성능이 나빠진다.

## 결정 (Decision)

**역할별로 식별자를 분리**한다 — 하나로 통일하지 않는다.

| 용도 | 식별자 | 이유 |
|------|--------|------|
| **엔티티 PK** | **UUID v7** (PostgreSQL native) | 시간 정렬 가능 → 삽입 지역성 확보(v4 단편화 회피) + 분산 생성 안전 |
| **공개 노출 ID** | **Base62 인코딩** | UUID를 짧고 URL 친화적으로. 순차 정수 노출 회피 |
| **임시 토큰·추적 ID**(`jti`, `request_id`) | **ULID 문자열** | 엔티티가 아닌 단명 식별자. 시간 정렬 + 문자열 친화 |

## 트레이드오프 (Consequences)

**얻은 것**: 순차 노출 차단 + 인덱스 지역성(v7) + 공개 ID 간결화 — 세 요구를 각각 최적 도구로.
**치른 비용**:
- UUID는 정수 PK보다 인덱스가 큼 — v7의 시간 정렬로 단편화 비용을 상쇄.
- 공개 ↔ 내부 인코딩 변환 비용(Base62) — 무시 가능 수준.
- 식별자가 3종이라 *일견 복잡* — 본 문서로 "역할 분리"임을 명시(혼용 아님).

## 고려한 대안 (Alternatives)

| 대안 | 기각 사유 |
|------|-----------|
| Auto-increment 정수 | 열거·추정 취약, 분산 생성 약점 |
| UUID v4 PK | 무작위 삽입 → 인덱스 단편화로 쓰기 성능 저하 |
| 단일 식별자로 통일 | PK(지역성)·공개ID(간결·은닉)·토큰(단명)의 요구가 상충 — 하나로는 최적화 불가 |

## 일부러 하지 않은 것 (Non-goals)

- Snowflake 등 **전역 분산 ID 서비스** 도입 — UUID v7로 충분하며 봉투 내 불필요한 운영 부담.
