# ADR 0013 — 제품 동작 결정 노트: 단일 세션·WS 토큰 전달·차단 시맨틱

- **상태**: 채택됨 (Accepted)
- **관련 코드**: `app/domain/auth/service.py`(`user_refresh:{user_id}` 단일 키),
  `app/domain/chat/ws_auth.py`(`?token=` 검증), `app/domain/chat/service.py`(`get_or_create_room`
  차단 검사), `app/domain/chat/router 계열`(room_id 기반 목록·기록·읽음)

## 맥락 (Context)

코드에 이미 구현된 동작이지만 "의도인지 우연인지"가 기록되지 않아, 후속 작업에서 버그로
오인되거나 무단으로 뒤집힐 수 있는 제품 결정 3건을 확정해 남긴다. 셋 다 아키텍처보다
**제품 동작**의 결정이라 개별 ADR 대신 한 노트로 묶는다.

## 결정 (Decision)

1. **Refresh 토큰 = 유저당 단일 키(단일 세션 정책)**
   - `user_refresh:{user_id}`에 다이제스트 하나만 저장(RTR 회전 대상도 이 키 하나).
     **두 번째 기기에서 로그인하면 첫 기기의 refresh가 무효**가 되어, access 만료 시점에
     첫 기기는 재로그인해야 한다.
   - 근거: 멀티 세션(기기별 refresh 목록)은 저장·회전·무효화 복잡도를 늘리는데, 이
     서비스의 봉투에서 다중 기기 동시 사용은 핵심 시나리오가 아니다. 부수 효과로 탈취된
     refresh는 정상 사용자의 다음 로그인에 자동 무효화된다(보안상 이점).
   - 안 한 선택: 기기별 refresh 세션 테이블(관리 UI·세션 목록 API까지 끌려오는 표면).

2. **WS 인증 토큰은 쿼리스트링(`?token=`)으로 전달**
   - 브라우저 WebSocket API는 커스텀 헤더를 지원하지 않아 표준적 대안이 쿼리스트링이다.
   - 노출 트레이드오프를 인지하고 수용한다: TLS로 전송 중 노출은 없지만 **접근 로그·프록시
     로그에 토큰이 남을 수 있다**. 완화: access 토큰은 수명이 짧고(만료 시 자동 무효),
     서버 접근 로그는 경로만 기록하며 쿼리스트링을 남기지 않는 구성을 전제한다.
   - 안 한 선택: 첫 프레임 인증(연결 후 토큰 프레임 전송) — 미인증 연결이 리소스를 먼저
     점유하는 창이 생기고 클라이언트 프로토콜이 복잡해진다. 티켓 발급(1회용 단기 티켓
     교환)은 봉투 대비 과잉.

3. **차단 시맨틱 = 비대칭(새 대화 차단, 과거 열람 허용)**
   - 차단 관계에서 `GET /chat/rooms/direct/{peer}`(방 열기/재개)는 403 — 새 메시지 발신은
     WS·REST 어느 진입점이든 `get_or_create_room` 깊이에서 거부된다.
   - 기존 room_id 기반 **목록·기록·읽음 처리는 계속 동작**한다(과거 대화 열람 허용).
   - 근거: 일반 메신저 관행(차단해도 과거 기록은 열람 가능)과 일치하고, 새 발신이 이미
     차단되므로 추가 데이터 노출이 없다. 열람 경로마다 차단 검사를 넣으면 쿼리 비용만
     늘고 사용자 가치가 없다.
   - **FE 계약**: direct-open 403은 "차단 관계" UX로 처리해야 한다(에러 문구는 누가
     차단했는지 노출하지 않는 중립 표현).

## 트레이드오프 (Consequences)

- 단일 세션: 다중 기기 사용자는 기기 전환 시 재로그인 마찰. 세션 목록·강제 로그아웃
  같은 관리 기능은 이 정책 위에 세울 수 없다(필요해지면 이 ADR을 갱신하고 구조 변경).
- WS 쿼리 토큰: 로그 위생(쿼리스트링 미기록)이 배포 환경 요건이 된다.
- 차단 비대칭: "차단하면 과거 대화도 사라져야 한다"는 기대를 가진 사용자에겐 어긋날 수
  있다 — 제품 요구가 분명해지면 열람 경로에 검사를 추가하는 방향으로 갱신.
