# 00 · 운영 전제와 범위 (Operating Envelope & Scope)

> **Phase 0 · Inception 산출물 — 확정 (Accepted, 2026-07-03).**
> 이 문서는 이후 모든 설계·ADR·복잡도 판정의 **단일 근거**다.
> 복잡도는 이 봉투 *안에서만* 정당화된다. 봉투 밖은 과잉으로 본다.

---

## 1. 목적

AI 기능이 아니라 **"현실적 트래픽을 가정한 운영 등급 백엔드 설계"**를 이 포트폴리오의 차별점으로 삼는다.
차별점의 증명은 *기능의 개수*가 아니라 **"각 복잡도가 명시된 운영 전제로 정당화된다"**는 판단력이다.
따라서 먼저 **전제(2)**를 못 박고, 그 위에서 **운영 과제(3)**를 분류하고, **재건 순서(4)**를 정한다.

---

## 2. 운영 전제 (Operating Envelope)

| 축 | 전제(확정) | 함의 |
|----|-----------|------|
| 규모 | 평상시 수천~수만 DAU | 단일 인스턴스로도 평시 처리 가능, 피크가 문제 |
| 핫스팟 | 인기글 1건에 **초당 수백~수천 조회** | 조회 경로가 최적화 1순위 (버퍼링·캐시) |
| 쓰기 | 신규 write는 초당 수십 규모 | 강한 정합성 필요한 write는 정직하게 트랜잭션으로 |
| 서버 | 멀티 인스턴스 **3~10대** (수평 확장) | 상태는 인스턴스 밖(Redis/DB)에. 로컬 상태 금지 |
| 배포 | 무중단 롤링 / 블루-그린 | 마이그레이션 하위호환, graceful shutdown 필요 |
| 가용성 | **99.9%** (월 ~43분 다운 허용) | 외부 I/O는 fallback 우선(가용성 > 순간 정합성) |

**상한 — 이걸 넘는 복잡도는 과잉으로 판정한다:**
멀티리전 · 금융권 강정합성 · 99.99%+ · 초당 수만 건 *신규 write* · exactly-once 배송.

## 4. 도메인 재건 순서 (의존도 기준)

앞 단계가 뒤 단계의 전제가 되도록 **의존도 낮은 기반부터** 재건한다(하이브리드: 검증된 로직은 이식).

1. **기반층(횡단)** — `db/`(세션·엔진), `core/config`(설정), `common/responses`(ApiResponse), `core/ids`(식별자), 로깅. → 3-A의 토대.
2. **auth / users** — 인증은 거의 모든 도메인의 전제.
3. **media** — 업로드는 posts·dogs·users가 의존.
4. **posts** — 핵심·최고부하(조회수·검색·페이지네이션). 가장 공들일 도메인.
5. **comments · likes** — posts에 의존.
6. **dogs** — users에 의존.
7. **chat · notifications** — 실시간/비동기.
8. **reports · admin** — 다른 도메인 데이터를 소비하는 관리 기능.
9. **Ops** — 배포·모니터링·로그 (Phase Ops).

각 도메인 재건 = Construction 단위. 단위마다 검증→계획(승인)→수정→검사→커밋.
